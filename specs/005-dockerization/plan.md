# Implementation Plan: Dockerization

**Branch**: `005-dockerization` | **Date**: 2026-06-17 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/005-dockerization/spec.md`

---

## Summary

Containerize all three Fe El Seka services — `services/api` (FastAPI + uv), `services/ai` (XGBoost FastAPI + uv), and `apps/main` (Next.js 14) — using multi-stage Dockerfiles, an nginx reverse proxy, development and production compose files, and a GitHub Actions `docker.yml` workflow that builds, vulnerability-scans (Trivy), and publishes images to GitHub Container Registry on merge to `main`. No new external services are introduced; Supabase continues to run via CLI locally and as a managed cloud service in production.

---

## Technical Context

**Language/Version**: Python 3.11 (FastAPI services) · TypeScript 5 / Node.js 20 (Next.js frontend)

**Package Managers**: uv (Python services — `services/api`, `services/ai`) · pnpm 8 (Next.js monorepo)

**Primary Dependencies**:
- `services/api`: FastAPI, asyncpg, GeoAlchemy2, resend, python-jose, pydantic-settings
- `services/ai`: FastAPI, XGBoost, scikit-learn, pandas, numpy, osmnx, supabase-py
- `apps/main`: Next.js 14, shadcn/ui, Tailwind CSS, Supabase JS, react-leaflet
- Infrastructure: nginx 1.27-alpine, Trivy (CI scanner), Docker Buildx, `docker/build-push-action@v5`

**Storage**: Supabase PostgreSQL (managed cloud / Supabase CLI local — not containerized)

**Testing**: Manual validation via quickstart scenarios; no new automated test framework introduced

**Target Platform**: Linux Docker containers (dev: macOS/Windows WSL2 · prod: Linux server)

**Project Type**: Monorepo — `apps/main` + `services/api` + `services/ai` + shared `packages/`

**Performance Goals**:
- Development stack cold start ≤ 60 seconds (SC-007)
- Hot reload reflects changes in ≤ 5 seconds (SC-005)
- CI pipeline (build + scan + push) completes in ≤ 10 minutes (SC-004)
- nginx response at `http://localhost/` ≤ 200ms (NFR-004)

**Constraints**:
- `apps/main/Dockerfile` build context must be monorepo root (`packages/` shared deps required)
- `output: 'standalone'` must be enabled in `next.config.mjs` before the frontend image can be built
- Both Python services use uv — Dockerfiles copy uv binary and use `uv sync --frozen --no-dev`
- No secrets in any image layer (SC-003) — all config injected via `env_file`
- All containers run as non-root (NFR-002)

**Scale/Scope**: MVP — single-server Docker Compose deployment; no horizontal scaling

## Implementation Notes for GLM

1. **uv binary copy pattern** — Do NOT install uv via pip or apt. Use `COPY --from=ghcr.io/astral-sh/uv:latest /uv /usr/local/bin/uv` in the builder stage. Run `uv sync --frozen --no-dev --no-editable`. The virtualenv lands at `/app/.venv`; copy it to the runner with `COPY --from=builder /app/.venv /app/.venv` and set `ENV PATH="/app/.venv/bin:$PATH"` and `ENV PYTHONUNBUFFERED=1`.

2. **Frontend build context is monorepo root** — In `docker-compose.yml` set `build.context: .` and `build.dockerfile: apps/main/Dockerfile`. In the Dockerfile, copy root-level `package.json`, `pnpm-lock.yaml`, `pnpm-workspace.yaml`, and `packages/` before copying `apps/main/`. Without this, `@fe-el-seka/ui` and `@fe-el-seka/shared` imports fail during `pnpm build`.

3. **`output: 'standalone'` is a prerequisite** — Add this to `apps/main/next.config.mjs` as task T001. Verify `pnpm --filter main build` produces `.next/standalone/server.js` before writing the Dockerfile.

4. **Health check endpoints** — Python services already have `/health` via `health_router`. Verify the endpoint returns `{"status":"ok"}`. Create `apps/main/src/app/api/health/route.ts` for the Next.js health check before enabling `HEALTHCHECK` in the frontend Dockerfile.

5. **nginx service names are DNS hostnames** — In the compose network, service names `api`, `ai`, `main` resolve as DNS. nginx `proxy_pass` must use `http://api:8000`, `http://ai:8000`, `http://main:3000` — never `localhost`.

6. **GHCR push is conditional on `main` only** — In `docker.yml`, set `push: ${{ github.ref == 'refs/heads/main' }}` on the `docker/build-push-action@v5` step. Build and scan run on both PR and push for early CVE feedback (FR-016).

7. **Trivy `--ignore-unfixed: false`** — Do not set `ignore-unfixed: true`. All CRITICAL/HIGH CVEs, whether fixable or not, must fail the pipeline (FR-015).

8. **`mem_reservation` uses Compose v2 `deploy` syntax**:
   ```yaml
   deploy:
     resources:
       reservations:
         memory: 256M
   ```
   Do not use the deprecated top-level `mem_reservation` key.

9. **Supabase CLI stays outside compose** — Do NOT add a `supabase` or `postgres` service to `docker-compose.yml`. Set `SUPABASE_URL=http://host.docker.internal:54321` in the api and ai `.env` files for local development.

10. **Root `.dockerignore` for frontend build** — Docker reads `.dockerignore` relative to the build context. Since `apps/main` uses `.` (root) as context, the exclusion file must be at the repository root as `.dockerignore`. It must exclude `node_modules/`, `**/.env*`, `.next/`, `specs/`, `docs/`, `**/*.test.*`, `.git/`.
