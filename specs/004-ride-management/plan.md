## Summary

Implement the driver-facing ride creation, editing, cancellation, and seat management system — the supply layer of the Fe El Seka platform. The backend (FastAPI) owns all business logic enforcing verified-driver gating, status lifecycle transitions, 2-hour overlap prevention, and verification-revocation-triggered auto-cancellation. The frontend (Next.js `apps/main`) provides a map-based ride creation form (Leaflet + OpenStreetMap) and a driver ride dashboard. PostGIS `geography` columns store origin/destination coordinates, GIST-indexed for Phase 5's spatial queries. A Supabase Database Webhook triggers automatic ride cancellation within 1 minute of driver verification revocation.

## Technical Context

**Language/Version**: Python 3.11 (FastAPI backend) · TypeScript 5 / Next.js 14 (frontend)

**Primary Dependencies**:
- Backend: FastAPI, SQLAlchemy (async), asyncpg, GeoAlchemy2, Resend (email)
- Frontend: Next.js 14, react-leaflet, Leaflet.js, shadcn/ui, Tailwind CSS, Supabase JS client

**Storage**: Supabase PostgreSQL with PostGIS (`geography(Point, 4326)` for coordinates)

**Testing**: pytest + HTTPX (backend API tests) · Vitest + React Testing Library (frontend unit tests)

**Target Platform**: Linux server (FastAPI) · Next.js SSR web app (mobile-first, driver role)

**Project Type**: Monorepo — `apps/main` (driver screens) + `backend/` (FastAPI API + services)

**Performance Goals**: Ride CRUD endpoints ≤500ms p95 · Dashboard render ≤2s for up to 200 rides

**Constraints**:
- All business rules enforced in FastAPI backend (Constitution §Architecture Standards)
- PostGIS `geography` types required for all coordinate fields (Constitution §Data Standards)
- RLS policies restrict drivers to their own ride data
- Verification enforcement is continuous, not one-time (FR-018)
- Email delivery is best-effort; cancellation must never be blocked by email failure (FR-021)

**Scale/Scope**: MVP — ≤1,000 concurrent users; expected tens of rides per day per driver at launch

## Implementation Notes for GLM

1. **Overlap check must use an advisory lock** — `pg_advisory_xact_lock(hashtext(driver_id::text))` inside the creation transaction before the `SELECT` that checks for conflicting departure times. Without this, two simultaneous POSTs from the same driver can both pass (see research.md §4).

2. **`available_seats` is a generated column** — never write to it directly. Only write `total_seats` and `booked_seats`. The DB always keeps it consistent (`total_seats - booked_seats`).

3. **Webhook secret validation** — the `/api/v1/internal/driver-revocation` endpoint must validate `X-Webhook-Secret` against the value stored in Supabase Vault before processing anything. Return `401` immediately on mismatch.

4. **Email retry sweep** — on FastAPI startup, launch a background `asyncio.Task` that calls `notification_service.retry_pending_emails()` every 5 minutes. Exponential backoff: 0 → 5m → 30m → 2h → 24h. After 5 retries, mark `failed_permanent`.

5. **GIST indexes** — create GIST indexes on `origin_coordinates` and `destination_coordinates` in the migration even though Phase 4 doesn't use them for spatial queries. Phase 5 will rely on these existing; re-indexing a large table later is disruptive.

6. **Nominatim rate limit** — in the Next.js app, debounce the reverse-geocode call by 300ms after the user drops a pin to avoid hammering Nominatim's 1 req/s limit.

7. **Status transitions in FastAPI** — use a transition table (dict) in `ride_service.py` to validate `(current_status, action)` pairs. Never accept arbitrary status values from the client. Return `ride_not_editable` for invalid transitions.
