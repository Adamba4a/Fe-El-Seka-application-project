# Implementation Plan: Authentication & Verification

**Branch**: `003-auth-verification` | **Date**: 2026-06-14 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/003-auth-verification/spec.md`

## Summary

Build the complete trust and identity layer for Fe El Seka: Supabase Auth phone OTP for passengers and drivers, email + password for admins, user profile creation with permanent role selection, identity document upload and manual admin review with a 3-attempt cap and support email lockout, vehicle registration for verified drivers, and an admin verification dashboard with session revocation on suspension.

## Technical Context

**Language/Version**: TypeScript 5.x (Next.js 14 frontend), Python 3.11+ (FastAPI backend)

**Primary Dependencies**:
- Frontend: `next@14`, `@supabase/supabase-js@2`, `@supabase/ssr`, `tailwindcss`, `shadcn/ui`, `react-hook-form`, `zod`
- Backend: `fastapi>=0.111`, `supabase>=2.4` (Python), `python-multipart`, `pydantic-settings>=2.2`, `python-jose[cryptography]`

**Storage**: Supabase PostgreSQL (existing from Phase 1), two new Supabase Storage buckets (`profile-photos`, `identity-documents`)

**Testing**: `pytest` + `httpx` (backend), manual browser testing (frontend flows per quickstart.md)

**Target Platform**: Web вЂ” mobile-first for `apps/main`; desktop-first for `apps/admin`

**Performance Goals**: Auth endpoints в‰¤500ms p95; admin verification queue renders in в‰¤2s for 500 items

**Constraints**: Session revocation must be immediate on suspension (two-layer: DB status check middleware + Supabase admin signOut); identity documents must never be publicly accessible; submission cap enforced at both frontend and backend

**Scale/Scope**: ~1,000 concurrent users; admin queue up to 500 pending items
